Why Git?

Version Control

Track history

Collaboration

Proposal evaluation

Backup purposes

Mostly suitable for plain-text files, such as source-code

Try to avoid putting (large) binary files in a repository

Distributed

Every user has their own copy of the repository

Changes can be shared directly between repositories

The (optional) server copy is not really special

  • Distributed versus centralized (such as CVS, Subversion)
  • Individual copies provide redundancy
  • A central repository can still be used, such as provided by GitHub
  • Server copy is typically 'bare'

Structure

  • Differences between centralized/distributed architecture
  • Closer look at the structure of a repository
  • Centralized version control uses a single master
  • Master has full history
  • Clients check out the current state into working copy
  • Changes from working copies are committed straight back to the master
  • Every client has the full history of all files it knows about
  • Clients check out a working copy from their own history
  • Changes are committed back to the local repository
  • Clients can share history using push and pull/fetch mechanics
  • Fetch retrieves history from another place
  • Pull retrieves history from another place and synchronizes it with local history
  • State is checked out from the version database
  • Changes are added to the stage
  • Staged changes are reset back to the working copy
  • Staged changes are comitted to the version database
  • Some 'exceptions to the rule' exist, e.g. new files being added to the stage are removed using git rm --cached

Git in Action

Create a repository

$ git init my-project && cd my-project
Initialized empty Git repository in /home/jkraak/Projects/my-project/.git/

Or clone an existing one

$ git clone git@github.com:bauglir/me-myself-and-git.git
Cloning into 'me-myself-and-git'...
remote: Counting objects: 7127, done.
remote: Total 7127 (delta 0), reused 0 (delta 0), pack-reused 7127
Receiving objects: 100% (7127/7127), 5.38 MiB | 2.52 MiB/s, done.
Resolving deltas: 100% (3767/3767), done.
Checking connectivity... done.

Add some files

$ echo 'foo' > foo.txt
$ echo 'bar' > bar.txt
$ git status
On branch master

Initial commit

Untracked files:
  (use "git add < file>..." to include in what will be committed)

          bar.txt
          foo.txt

nothing added to commit but untracked files present (use "git add" to track)

git status is your friend! It provides hints on what to do next 

$ git add foo.txt
$ git status
On branch master

Initial commit

Changes to be committed:
  (use "git rm --cached < file>..." to unstage)

        new file:   foo.txt
        new file:   bar.txt
$ git commit -m 'Add foo and bar literature'
[master (root-commit) 3f91579] Add foo and bar literature
 2 files changed, 2 insertions(+)
 create mode 100644 bar.txt
 create mode 100644 foo.txt

Typically the commit message is not directly entered at the command line. When leaving off the -m flag your default editor will be opened to provide a commit message.

Note the hashes, we'll talk about those later

Making Changes

$ echo 'baz' >> foo.txt
On branch master
Changes not staged for commit:
  (use "git add < file>..." to update what will be committed)
  (use "git checkout -- < file>..." to discard changes in working directory)

        modified:   foo.txt

no changes added to commit (use "git add" and/or "git commit -a")
$ git diff
diff --git a/foo.txt b/foo.txt
index 257cc56..0c071e1 100644
--- a/foo.txt
+++ b/foo.txt
@@ -1 +1,2 @@
 foo
 +baz
$ git add -u && git commit -m 'Add baz reference'
[master e981856] Add baz reference
 1 file changed, 1 insertion(+)

Note the -u which only adds changes to files already being tracked

By repeating this process the history is slowly built up
