Why Git?

Version Control

Track history

Collaboration

Proposal evaluation

Backup purposes

Mostly suitable for plain-text files, such as source-code

Try to avoid putting (large) binary files in a repository

Distributed

Every user has their own copy of the repository

Changes can be shared directly between repositories

The (optional) server copy is not really special

  • Distributed versus centralized (such as CVS, Subversion)
  • Individual copies provide redundancy
  • A central repository can still be used, such as provided by GitHub
  • Server copy is typically 'bare'

Structure

  • Differences between centralized/distributed architecture
  • Closer look at the structure of a repository
  • Centralized version control uses a single master
  • Master has full history
  • Clients check out the current state into working copy
  • Changes from working copies are committed straight back to the master
  • Every client has the full history of all files it knows about
  • Clients check out a working copy from their own history
  • Changes are committed back to the local repository
  • Clients can share history using push and pull/fetch mechanics
  • Fetch retrieves history from another place
  • Pull retrieves history from another place and synchronizes it with local history
  • State is checked out from the version database
  • Changes are added to the stage
  • Staged changes are reset back to the working copy
  • Staged changes are comitted to the version database
